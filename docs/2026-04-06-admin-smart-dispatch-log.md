# 2026-04-06 管理端智能分配归档视图开发记录

## 1. 本轮目标

在已经完成：

- 患者端智能分配进度展示
- 医生工作台“系统推荐给我”待办
- 医生问诊列表智能分配筛选与详情卡片

之后，继续把“智能分配”补到管理端，让运营和产品在后台也能直接看到：

- 当前有多少问诊还在等待首推医生接手
- 有多少问诊已经由首推医生承接
- 某一条问诊最终是由谁接手、系统当时推荐了谁
- 当前问诊是处于首推等待、首推接手，还是被其他医生接手

这一轮的重点仍然不是新增新的派单算法，而是把已有推荐与认领数据整理成统一的后台归档视图。

## 2. 后端实现

更新文件：

- `template-backend/src/main/java/cn/gugufish/service/impl/ConsultationRecordAdminServiceImpl.java`

本轮继续复用已有智能分配聚合工具：

- `template-backend/src/main/java/cn/gugufish/utils/ConsultationSmartDispatchUtils.java`

### 2.1 管理端列表补充智能分配摘要

在 `listRecords()` 中，为每条问诊记录补充：

- 当前认领医生
- 当前认领状态
- 分诊结果中的首推医生
- 候选医生信息
- 推荐依据说明

最终统一通过 `ConsultationSmartDispatchUtils.build(...)` 生成 `smartDispatch` 字段，直接挂到管理端列表响应里。

这样管理端在列表页就能看到：

- 这条问诊当前是否还在等待首推医生
- 是否已经由首推医生承接
- 是否被其他医生接手

### 2.2 管理端详情补充智能分配归档

在 `recordDetail()` 中同样补充 `smartDispatch`，让管理端详情弹窗可以完整回看：

- 系统首推医生是谁
- 当时推荐依据是什么
- 候选医生大概有多少
- 当前实际是谁接手

这一层没有新增数据库表，也没有改变当前认领链路，只是把既有分诊和认领结果做了统一聚合。

## 3. 前端实现

更新页面：

- `template-front/src/views/admin/ConsultationRecordPage.vue`

### 3.1 列表页新增智能分配统计卡

顶部统计新增：

- `等待首推医生`
- `首推已接手`

这样进入管理端问诊记录页后，就能先看到当前智能分配的承接情况，而不需要逐条翻列表。

### 3.2 工具栏新增智能分配筛选

工具栏新增 `dispatchFilter`，支持筛选：

- 等待首推医生
- 首推医生已接手
- 其他医生接手
- 科室分诊队列

这样运营排查“推荐有没有被真正接住”时，会比单纯看认领状态更直接。

### 3.3 列表新增智能分配列

问诊记录表格新增“智能分配”列，展示：

- 智能分配状态标签
- 首推医生或当前接手医生的摘要文案

前端继续复用共享的 `dispatch.js` 归一化逻辑，避免患者端、医生端、管理端三处各自维护一套状态解释。

### 3.4 详情弹窗新增智能分配归档卡片

详情弹窗新增“智能分配归档”区块，补充展示：

- 当前智能分配状态
- 首推医生
- 候选医生数量
- 推荐依据
- 排班参考

到这里，管理端查看单条问诊时，已经可以比较完整地回看“系统如何推荐、最终如何承接”。

## 6. 当前阶段结论

到这里，“智能分配”已经不再只是患者端和医生端可见，管理端也具备了统一的归档和筛选能力：

1. 患者能看到当前推荐与接手进度
2. 医生能看到系统推荐给自己的待办
3. 管理端能看到推荐是否真正被首推医生承接
4. 单条问诊可以回看推荐依据与实际接手结果

这让后续继续推进：

- 智能分配接单率分析
- 首推命中率分析
- 自动分配策略实验

会更顺畅，因为三端都已经有统一的状态视图。

## 7. 下一步建议

建议下一步优先继续补下面几个方向：

1. 在管理端增加“首推命中率 / 被其他医生接手率 / 平均等待接手时长”等运营指标
2. 为智能分配加入容量、排班、风险级别等更细的优先级策略
3. 为后续自动分配预留实验开关和策略参数配置
